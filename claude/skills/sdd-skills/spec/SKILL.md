# /spec — Spec 驱动的需求全流程分析

## 概述

一次性完成工作区初始化、需求理解、方案设计、测试用例设计，产出核心文档：

| 文档 | 回答什么 | 核心内容 | 谁来读 |
|------|---------|---------|--------|
| **spec.md** | 做什么 | 需求目标、风险评估、追溯矩阵（6 章全景，第 2/3 章引用独立文件） | 所有人 |
| **contracts.md** | 接口契约 | API 定义、入参出参、错误码、请求响应示例 | 开发者、评审人 |
| **testcases.md** | 怎么验 | 验收标准(AC)、测试用例(TC) | 开发者、测试、/testing |
| **design.md** | 怎么做 | 方案决策、对比评分、自审核、实施代码 | 架构师、评审人、开发者 |
| **tasks.md** | 具体步骤 | 任务清单、文件变更、构建顺序、回滚策略、Context Brief | 开发者、/develop |
| **solution-design.html** | 可视化总览 | 整合 spec+design+tasks 精华，带样式的完整方案设计文档 | 评审会议、PM、存档 |
| **repos.md** | 代码在哪 | 仓库 worktree 清单、项目上下文、部署配置 | /develop、/testing |
| **CLAUDE.md** | 上下文恢复 | 流水线状态、关键决策、快速开始 | 新会话冷启动 |

核心文档通过 **ID 系统**互相引用：`G-xxx`（需求）、`NFR-xxx`（非功能需求）、`DEC-xxx`（决策）、`API-xxx`（接口）、`AC-xxx`（验收标准）、`TC-xxx`（测试）、`R-xxx`（风险）、`TASK-xxx`（任务），形成完整追溯链。

**核心理念：**
1. **Spec 即契约** — spec.md 定义"做什么"，后续 /develop、/review、/testing 全部以此为准
2. **TDD 前置** — 测试用例在开发前生成，开发时对着测试写代码
3. **SDD 开发** — 接口契约先行，开发者对着契约编码
4. **方案必须基于代码现状** — 不做脱离代码的空中楼阁设计
5. **关注点分离** — 需求(spec)、方案(design)、执行(tasks)各司其职

## 参数

- 必填：需求描述（文字、文档路径或 URL）
- 可选：`--review` — 追加多视角评审（架构师 / 前端UX / DBA运维）
- 可选：`--template {模板名}` — 基于已有模板快速生成
- 可选：`--templates` — 查看可用模板列表

---

## 铁律

```
1. 任何需求都不允许带着未回答的问题离开这个阶段
2. 每个决策点必须有 ≥2 个候选方案的对比，不允许直接给结论
3. 方案设计必须基于代码探索，确认技术可行性
4. 非功能需求（性能/安全/兼容）必须显式声明，不能只有功能目标
5. 验收标准（AC）必须在测试用例之前定义，用 Given/When/Then 格式
6. 测试用例必须在开发前生成（TDD 前置），关联 AC 而非直接关联 G
7. 风险必须显式评估并制定缓解措施，不能只靠自审 checklist
8. 所有文档通过 ID 系统互相引用，追溯矩阵覆盖率 100%
9. 文件变更清单必须精确到文件级
10. 每个方案完成后必须执行自审核清单
11. Worktree 必须创建成功后才能进入设计阶段
12. 涉及数据库变更时，必须通过 subagent 验证表结构和字段分布
```

### Phase 2：代码工作区准备 → repos.md

**目标：** 确定受影响的代码仓库，创建 worktree 工作区，收集部署配置。

> **关键概念区分：**
> - **Git 分组/命名空间**（如 `EDU_zhixue_pt`）— 这是 GitLab/组织层面的分组，**不是** git 仓库
> - **实际 git 仓库**（如 `xqr_report/srv-pt-report-qyjy`）— 包含 `.git` 目录，可以执行 git 操作
>
> 仓库路径可能是多级的（如 `group/subgroup/repo-name`），不要把顶层分组名当作仓库。

#### 2.1 识别代码仓库

1. **了解仓库结构** — 读取 `~/knowledge-base/overview.md` 的"代码仓库结构"章节，了解各 git 仓库包含哪些项目。注意区分分组层级和实际仓库层级
2. **交叉定位仓库** — 根据 Phase 1 的需求分析结果，按 模块 → 项目 → **实际 git 仓库** 的链路，定位受影响的仓库。仓库路径是从代码根路径到 `.git` 所在目录的相对路径
3. **向用户确认仓库列表** — 展示受影响的仓库列表（包含完整仓库路径），请用户确认或调整

#### 2.2 创建 worktree

1. **获取代码根路径** — 首次使用时询问用户本地代码根目录（如 `/Users/xxx/code/`），后续从 `repos.md` 中读取复用
2. **确认 feature 分支名** — 向用户确认 feature 分支名（默认建议 `feature/{功能名缩写}`）
3. **验证、同步并创建 worktree** — 对每个确认的仓库：

```bash
# 1. 验证目标确实是 git 仓库
git -C {代码根路径}/{仓库路径} rev-parse --git-dir

# 2. 拉取远程最新代码，确保基于最新的主分支创建 feature 分支
git -C {代码根路径}/{仓库路径} fetch origin
git -C {代码根路径}/{仓库路径} pull origin main

# 3. 创建 worktree（本地目录名取仓库路径的最后一段）
git -C {代码根路径}/{仓库路径} worktree add {CWD}/{仓库目录名} -b {feature分支名}
```

> **注意：** 步骤 2 中的 `main` 应替换为仓库实际的主分支名（可能是 `master`、`develop` 等），可通过 `git -C ... symbolic-ref refs/remotes/origin/HEAD` 自动检测。如果 pull 遇到冲突或本地有未提交的变更，必须先解决后再继续。

> **命名规则：**
> - `{仓库路径}`：从代码根路径到 git 仓库的完整相对路径，如 `xqr_report/srv-pt-report-qyjy`
> - `{仓库目录名}`：worktree 在 CWD 下的目录名，取仓库路径的最后一段，如 `srv-pt-report-qyjy`
> - 如果多个仓库的最后一段重名，加前缀区分，如 `xqr-srv-pt-report-qyjy`

#### 2.3 收集部署配置

对每个 worktree，扫描其下各项目目录的 `CLAUDE.md`，提取部署配置信息：

1. 对每个 worktree，扫描其下各项目目录的 `CLAUDE.md`，提取 `流水线地址`（质效地址）和 `git地址`
2. 项目名从目录名或 README.md 中提取
3. env 默认为 `test`
4. 将提取结果以表格形式展示给用户确认：

```markdown
从项目 CLAUDE.md 中提取到以下部署配置，请确认或补充：

| 项目名 | Git 地址 | 质效地址 | 环境 |
|--------|---------|---------|------|
| {项目名} | {git地址} | {质效地址} | test |
```

5. 用户确认后（可修改/补充缺失字段），写入 `repos.md`

#### 2.4 写入 repos.md

将 worktree 清单、项目上下文和部署配置统一写入 `.ai-workspace/repos.md`：

```markdown
# 代码仓库工作区

## 基本信息
- 代码根路径：{代码根路径}
- Feature 分支：{feature分支名}
- 技术栈：{从项目文件自动检测}

## Worktree 清单
| 仓库路径 | 源仓库路径 | Worktree 目录名 | Worktree 路径（实际编码路径） | Feature 分支 | 涉及的项目 |
|----------|-----------|----------------|---------------------------|-------------|-----------|
| {仓库路径} | {代码根路径}/{仓库路径} | {仓库目录名} | {CWD}/{仓库目录名} | {feature分支名} | {项目列表} |

## 部署配置
| 项目名 | Git 地址 | 质效地址 | 环境 |
|--------|---------|---------|------|
| {项目名} | {git地址} | {质效地址} | {env} |
```

**★ HARD-GATE：所有 worktree 创建成功。** 如果任何 worktree 创建失败（如路径不是 git 仓库、分支冲突等），必须先解决问题。

### Phase 3：设计方案 → design.md

**目标：** 基于代码现状设计可落地的方案，多方案对比评分。

#### 3.0 数据库结构探索（按需）

当需求涉及**数据库查询变更、数据模型修改、报表逻辑调整、数据过滤/聚合条件变更**时，需要查询实际数据库以补充代码探索无法获取的信息。

> **触发条件判断：** 如果从代码中发现了 SQL/Mapper 变更、表结构依赖、字段过滤条件等数据层面的需求，则执行本步骤。如果需求纯粹是前端/接口/配置层变更，跳过。

**上下文保护原则：** 数据库的 DDL、表结构、数据样本可能非常庞大，**严禁直接在主会话中执行数据库查询**。所有数据库探索必须通过 **subagent（Agent 工具）** 完成，subagent 负责：
1. 执行查询
2. 分析结果
3. **仅将精炼后的结论**返回主会话

**执行流程：**

1. **识别待查询的数据库和表** — 从需求和代码中提取涉及的数据库类型、表名、需要确认的字段名
2. **确认数据库连接信息** — 检查以下来源（按优先级）：Auto Memory → `~/knowledge-base/` → 项目配置文件 → 询问用户
3. **分派 subagent 执行查询** — 为每个需要探索的数据库分派独立的 subagent：

```
你是一个数据库结构探索助手。请使用 /db-use 技能连接数据库并执行以下查询，
然后将结果提炼为简洁的摘要返回。

数据库连接信息：{连接信息}

请执行以下查询并总结：
1. 表结构查询（DDL/DESCRIBE）
2. 字段数据分布（GROUP BY + COUNT，仅查询与需求相关的字段）
3. 索引/分区信息（如果是 ClickHouse）

请返回摘要格式（不要返回原始查询结果）：
### 表结构摘要
### 字段数据分布
### 索引与性能
### 发现的问题
```

> **按需查询原则：** 只查询与需求直接相关的表和字段，不要全库扫描。如果 ~/knowledge-base/ 中已有表结构文档且足够新，可以跳过 DDL 查询。

4. **整合发现** — 将关键发现记录为设计素材，问题标记为需向用户确认

**跳过条件：** 需求不涉及数据库变更 / ~/knowledge-base/ 中已有完整的近期表结构文档 / 用户明确表示不需要

#### 3.1 代码探索（不可跳过）

启动 Explore Agent 深入分析受影响的代码模块：
- 追踪核心调用链路（入口 → 业务逻辑 → 数据层）
- 标注每个环节的关键代码位置（文件:行号）
- 确认运行环境和框架约束
- 识别受影响的文件和模块

#### 3.2 方案对比

识别关键决策点（DEC-xxx），对每个决策点提出 2-3 个候选方案：

| 要求 | 说明 |
|------|------|
| ≥2 个候选方案 | 至少 1 个保守方案 + 1 个进取方案 |
| 5 维度评分 | 每项 1-5 分，满分 25 |
| 推荐理由 | 核心理由不超过 3 点 + 排除理由 |
| 实施代码 | 关键改造代码 + 改造前后对比 |

**5 维评分体系：**

| 维度 | 问的是什么 |
|------|----------|
| 确定性 | 方案成熟吗？收益能量化吗？ |
| 可控性 | 改动范围清楚吗？过程中会不会越改越大？ |
| 可逆性 | 出问题能回滚吗？有降级策略吗？ |
| 运维友好 | 上线后好监控、好排查吗？ |
| 生态兼容 | 和现有技术栈融合吗？ |

#### 3.3 方案自审（内置，不可跳过）

```
□ 每个改动点是否有回退路径？
□ 是否引入了不必要的复杂性？能用现有基础设施解决吗？
□ 是否确认了运行环境（框架版本、容器类型）？
□ 多服务部署顺序依赖是否明确？
□ SQL/数据变更是否可回滚？
□ 接口变更是否向后兼容？不兼容时调用方是否已识别？
□ 并发场景是否安全？
```

**★ HARD-GATE：用户确认方案**

### Phase 4：定义契约与测试 → contracts.md, testcases.md, spec.md 第 2-4 章

**目标：** 从需求目标 + 方案设计自动派生接口契约、验收标准、测试用例，并评估风险。

#### 4.1 接口契约（SDD 核心）

为选定方案定义接口契约（API-xxx），每个接口包含：
- 服务名、方法名
- 入参（类型、是否必填、校验规则）
- 出参（类型、结构）
- 行为描述（逻辑说明，含边界处理）
- **错误码**（错误码、含义、触发条件）
- **请求/响应示例**
- **幂等性和版本兼容说明**

#### 4.2 验收标准（AC）

从 G-xxx 和 NFR-xxx 派生验收标准（AC-xxx），使用 Given/When/Then 格式：

```
AC-001: Given 数学科组长登录, When 访问数学科目, Then 正常显示
AC-002: Given 数学科组长登录, When 访问英语科目, Then 拒绝访问
```

**验收标准是需求和测试之间的桥梁。** 产品经理看得懂 AC，确认 AC 覆盖了预期场景，再从 AC 派生 TC。

#### 4.3 测试用例设计（TDD 前置）

从验收标准(AC-xxx) + 接口契约(API-xxx) 自动派生测试用例(TC-xxx)，写入 testcases.md。

**分析步骤：**
1. 从 AC-xxx 提取用户流程、UI 状态、数据字段、业务规则
2. 从 API-xxx 提取接口端点、入参出参、错误码，识别边界情况
3. 交叉验证：识别需求与方案之间的差异
4. 功能分解：将复杂功能分解为最小可测试单元
5. 场景拆解：将复杂需求拆解为具体的使用场景
6. 路径覆盖：确保覆盖所有可能的业务路径和分支
7. 综合应用下方所有测试设计方法以最大化覆盖率

**用例字段规范（9 个字段，全部必填）：**

| 字段 | 说明 | 要求 |
|------|------|------|
| ID | 用例编号 | 格式 `TC-{模块缩写}-{序号}`，补零至4位 |
| 标题 | 主谓宾结构 | 适合自动化的用例标题后添加"(自动化)"后缀 |
| 优先级 | P1-P5 | 见优先级定义 |
| 测试项 | 被测对象 | 限制10个中文汉字以内 |
| 前置条件 | 执行前必须满足的条件 | 无特殊条件填"无" |
| 步骤 | 操作序列 | 格式：`1.操作A 2.操作B 3.操作C` |
| 预期结果 | 与步骤一一对应 | 格式：`1.结果A 2.结果B 3.结果C` |
| 测试类型 | 功能/兼容性/易用性/安全性/可靠性/数据/集成/稳定性/移动端/埋点 | 从列表选择 |
| 是否反向用例 | 是/否 | 只允许两个值 |

**优先级定义：**

| 级别 | 定义 | 目标占比 |
|------|------|---------|
| P1 | 核心功能的正向测试用例（冒烟用例） | 10% |
| P2 | 基本功能的正向测试用例 | 40% |
| P3 | 核心功能的反向测试用例（重要的错误和边界值） | 20% |
| P4 | 基本功能的反向测试用例（异常测试、边界） | 20% |
| P5 | 不常用功能的正反向测试用例 | 10% |

**数量要求：** 每个功能点至少 8-15 条，正向与反向比例约 4:6。

**测试设计方法：**

核心技术（强制应用）：
1. **等价类划分** — 有效/无效等价类，每类至少1个用例
2. **边界值分析** — 边界值±1、最小值、最大值、空值、null
3. **因果图法** — 输入条件与输出结果的因果关系
4. **决策表法** — 多条件组合的复杂逻辑
5. **状态转换图** — 所有状态转换路径
6. **场景法** — 主流程和备选流程

扩展技术（建议应用）：错误推测法、流程分析法、数据驱动测试、基于风险的测试、正交试验法、配对测试法、探索性测试法、基于模型的测试、用户故事测试法

**覆盖维度：**
- **输入覆盖**：数据类型、长度、格式、内容
- **业务逻辑覆盖**：权限、状态、规则、流程
- **环境覆盖**：浏览器、设备、网络、数据量、并发操作

**质量规则：**
1. 可追溯性：每个用例必须关联 AC-xxx
2. 独立性：每个用例独立可执行
3. 完整性：9 个字段全填，不允许为空
4. 自动化标注：适合自动化的用例标题后加"(自动化)"
5. 安全测试专项：涉及权限的模块须包含注入、XSS、越权等测试
6. 数据安全测试：涉及数据传输/存储须覆盖加密和传输安全

NFR 对应的测试用例使用对应的测试类型（安全性测试、稳定性测试等）。

#### 4.4 风险评估

识别方案中的风险（R-xxx），评估概率和影响，制定缓解措施：

| 字段 | 说明 |
|------|------|
| ID | R-xxx |
| 风险描述 | 可能出什么问题 |
| 概率 | 高/中/低 |
| 影响 | 高/中/低 |
| 缓解措施 | 怎么降低风险 |
| Owner | 谁负责缓解 |

### Phase 5：规划任务 → tasks.md

**目标：** 从方案和接口契约派生可执行的任务清单。

1. 从 design.md 的影响文件生成文件变更清单：
   - **must-change**：必须修改的文件（附变更内容、关联 TASK-xxx）
   - **no-change**：确认不改的文件（附排除原因）
   - **pending**：待确认的文件

2. 派生任务（TASK-xxx），每个任务关联：
   - `test_ref`：完成后哪些 TC-xxx 应该通过
   - `api_ref`：实现哪个 API-xxx
   - `risk_ref`：关联哪些 R-xxx（风险缓解要求写入 Context Brief）
   - 模块名称
   - 依赖关系（depends_on）
   - **工作目录**（必须是 worktree 路径，从 repos.md 获取）
   - **Context Brief**（自足上下文，供 /develop Teammate 冷启动）

3. 确定构建部署顺序

4. 制定回滚策略（代码回滚 + 数据回滚 + 特性开关）

5. Worktree 映射（将任务关联到 Phase 2 已创建的 worktree，引用 repos.md 中的路径）

6. 定义成功指标（上线后怎么验证功能达到预期效果）

**★ HARD-GATE：用户确认任务规划**

**HTML 结构要求：**
- **Header**：渐变背景，需求标题 + 元信息（ID、优先级、作者、日期、状态）

  1. 需求概览（G-xxx 目标表 + NFR + 非目标 + 约束）
  2. 现状分析（调用链路流程图 + 问题汇总）

## 危险信号

**绝不要：**
- 在没有用户确认的情况下产出文档
- 跳过方案对比直接给结论
- 不探索代码就设计方案
- 遗漏测试用例设计
- 猜测不确定的业务逻辑
- 文档之间 ID 引用不一致
- 跳过 worktree 创建直接进入设计阶段
- 需求涉及数据库变更，但没有通过 subagent 验证表结构和字段分布
- 在主会话中直接执行数据库查询（必须用 subagent）

**始终要：**
- 每个决策有 ≥2 个候选方案
- 方案基于代码探索，确认技术可行性
- 测试用例在开发前生成
- 追溯矩阵覆盖率 100%
- 接口契约精确到入参出参行为
- 执行方案自审核清单
- 确保 worktree 创建成功后再进入设计阶段
- 产出完成后生成 CLAUDE.md 供新会话恢复上下文

## 常见自我合理化

| 借口 | 现实 |
|------|------|
| "这个方案明显最优" | 明显最优说明没考虑足够多维度 |
| "代码不用看，从文档就知道" | 文档可能过时，代码才是事实 |
| "评审太浪费时间" | 评审发现的 bug 在开发后修成本 10 倍 |
| "代码示例写伪代码就行" | 伪代码掩盖了真正的技术难点 |
| "这个改动太简单不需要方案" | 简单改动出的 bug 往往最难定位 |
| "部署顺序无所谓" | 分布式系统最怕部署顺序问题 |
| "从代码里看字段名就够了，不用查库" | 代码里的字段名不能告诉你数据分布和 NULL 情况 |
| "worktree 后面再创建也行" | 设计阶段就需要代码探索，没有 worktree 无法准确定位 |
